# 美容室用画像変換アプリ 要件定義書

**作成日**: 2025年10月30日  
**バージョン**: 1.0  
**プロジェクト名**: 美容室画像変換システム

---

## 1. プロジェクト概要

### 1.1 目的
美容室サイトに掲載するモデル画像やスタイリスト画像を、AI技術を活用して魅力的な画像に変換するWebアプリケーションを開発する。直感的なUI/UXにより、技術的知識がないユーザーでも簡単に高品質な画像変換が可能となることを目指す。

### 1.2 技術スタック
- **言語**: Python 3.11+
- **フレームワーク**: Django 5.0+
- **画像変換API**: Google Gemini 2.5 Flash Image (Vertex AI経由)
- **インフラ**: Docker + VPS
- **プロトコル**: HTTPS必須

### 1.3 システムの特徴
- クローズドシステム（管理者発行の認証情報でのみアクセス可能）
- 月間利用制限による公平な利用管理
- リアルタイムの処理進捗表示
- 30日間の画像保管とギャラリー機能

## 2. 機能要件

### 2.1 認証・ユーザー管理機能

#### 2.1.1 ログイン機能
**概要**: 管理者が発行したID/PASSでのみアクセス可能なクローズドシステム

**詳細仕様**:
- **ログイン画面**
  - 入力項目: ユーザー名、パスワード
  - ログインボタン押下で認証処理実行
  - 認証成功時: メイン画面（画像アップロード画面）へリダイレクト
  - 認証失敗時: エラーメッセージ表示「ユーザー名またはパスワードが正しくありません」
  - ログイン失敗によるアカウントロック機能は実装しない
  
- **セッション管理**
  - ログイン状態の保持（Djangoの標準セッション機能を使用）
  - 一定時間操作がない場合の自動ログアウト（推奨: 24時間）
  - ログアウト機能（全画面のヘッダーに配置）

#### 2.1.2 ユーザー情報
**保持する情報**:
- ユーザー名（ログインID、英数字、一意制約）
- パスワード（ハッシュ化して保存、8文字以上）
- 月間利用可能回数（デフォルト100枚、管理者が個別に変更可能）
- 当月の利用済み回数
- アカウント作成日時
- 最終ログイン日時

**パスワードポリシー**:
- 最小8文字以上
- ユーザー自身によるパスワード変更機能は不要
- パスワードは管理者が管理画面で指定

### 2.2 管理者機能

#### 2.2.1 管理画面へのアクセス
**概要**: Django標準のadminインターフェースを活用した管理機能

**アクセス方法**:
- `/admin/` エンドポイントからアクセス
- スーパーユーザー権限を持つアカウントでログイン

#### 2.2.2 ユーザー管理機能

**2.2.2.1 ユーザーの新規作成**
- **入力項目**:
  - ユーザー名（必須、一意）
  - パスワード（必須、8文字以上）
  - 月間利用可能回数（デフォルト100枚）
- **作成フロー**:
  1. 管理画面で「ユーザー追加」ボタンをクリック
  2. 上記項目を入力
  3. 保存ボタンでユーザー作成完了
  4. 作成したユーザー名とパスワードを利用者へ伝達（アプリ外での運用）

**2.2.2.2 ユーザーの削除**
- 管理画面のユーザー一覧から削除対象を選択
- 削除実行（論理削除または物理削除、推奨は論理削除）
- 削除されたユーザーはログイン不可となる

**2.2.2.3 利用回数制限の増減**
- **操作方法**:
  - ユーザー詳細画面で「月間利用可能回数」フィールドを編集
  - 任意の数値を設定可能（0〜上限なし）
  - 保存後、即座に反映
- **使用例**:
  - 特定ユーザーの月間制限を100枚→200枚に増加
  - テストユーザーの制限を無制限（例: 9999枚）に設定

#### 2.2.3 利用履歴確認機能

**2.2.3.1 ユーザーごとの利用履歴**
- **表示項目**:
  - ユーザー名
  - 当月の利用済み回数 / 利用可能回数
  - 利用履歴一覧（日時、元画像名、生成枚数、ステータス）
- **フィルタリング機能**:
  - 月別での絞り込み
  - ユーザー名での検索
- **詳細情報**:
  - 各画像変換処理の詳細（使用したプロンプト、生成時刻、処理時間など）

### 2.3 画像アップロード機能

#### 2.3.1 フォルダからのアップロード（初期リリース実装）

**2.3.1.1 基本仕様**
- **対応形式**: JPEG (.jpg, .jpeg), PNG (.png), WebP (.webp), HEIC/HEIF (.heic, .heif) ※HEIC/HEIF はアップロード時にJPEGへ変換して保存
- **ファイルサイズ制限**: 1ファイルあたり最大10MB
- **同時アップロード**: 最大10ファイルまで（推奨値）

**2.3.1.2 アップロード方法**
1. **ドラッグ&ドロップ**
   - 画面中央のドロップゾーンに画像ファイルをドラッグ
   - 複数ファイルの同時ドロップ対応
   - ドロップ時の視覚的フィードバック（エリアのハイライト表示）

2. **ファイル選択ダイアログ**
   - 「ファイルを選択」ボタンをクリック
   - OSのファイル選択ダイアログが開く
   - 複数選択可能（Ctrl/Cmdキー + クリック）

**2.3.1.3 アップロード後の処理**
- サムネイル表示（アップロードした画像のプレビュー）
- ファイル名の表示
- 削除ボタン（アップロード取り消し）
- バリデーション:
  - 非対応形式の場合: エラーメッセージ「対応していないファイル形式です」
  - サイズ超過の場合: エラーメッセージ「ファイルサイズは10MB以下にしてください」

#### 2.3.2 HotPepperBeauty URL取得機能（将来実装予定）

**機能概要**: HotPepperBeautyのスタイル/スタイリスト/ブログページのURLを入力すると、ページ内の画像を自動取得してアップロード

**詳細仕様（参考）**:
- **入力項目**: HotPepperBeauty のページURL
- **対象ページ**:
  - スタイルページ（ヘアスタイル写真）
  - スタイリストページ（スタッフ写真）
  - ブログページ（投稿画像）
- **取得処理**:
  - URLからHTMLを取得
  - 画像URLを抽出（スクレイピング）
  - 画像をダウンロードしてシステムに保存
  - 自サロンのURLのみ利用を想定（利用規約確認済み）
- **取得後の動作**:
  - 通常のアップロード画像と同様に扱う
  - サムネイル表示、削除可能

**注意事項**: 初期リリースには含まず、将来のアップデートで実装予定

### 2.4 画像変換機能

#### 2.4.1 プロンプト入力機能

**2.4.1.1 ワンタッププロンプト（プリセット）**
- **目的**: 専門知識がなくても効果的な変換を実現
- **プリセット内容**:
  1. **バックスタイル生成**: 「正面写真から背面のヘアスタイルを生成してください」
  2. **シンプル背景生成**: 「背景を白い無地の背景に変更してください」
  3. **髪の艶感増加**: 「髪に自然な艶と輝きを追加し、健康的な印象を強調してください」
  4. **明るく爽やかな雰囲気**: 「全体の色調を明るくし、爽やかで清潔感のある雰囲気に仕上げてください」
  5. **プロフェッショナルな印象**: 「プロフェッショナルで洗練された印象の画像に変換してください」

- **UI**:
  - プリセットボタンを横並びまたはドロップダウンで配置
  - ボタンクリックで自動的にプロンプトが入力される
  - 選択中のプロンプトを視覚的に表示（ハイライト等）

**2.4.1.2 カスタムプロンプト入力**
- テキストエリアでユーザーが自由にプロンプトを記述可能
- 最大文字数: 500文字（推奨）
- プレースホルダー: 「どのように変換したいか説明してください（例: 背景を明るくして、髪に艶を出す）」
- ユーザー独自のプロンプト保存機能は不要

#### 2.4.2 複数枚生成機能

**2.4.2.1 生成枚数の指定**
- **入力方法**: ドロップダウンまたは数値入力フィールド
- **選択肢**: 1枚、2枚、3枚、4枚、5枚
- **デフォルト値**: 3枚（推奨）
- **最大値**: 5枚

**2.4.2.2 生成ロジック**
- 同じプロンプトで指定枚数分の画像を生成
- Gemini API を生成枚数分呼び出し（各呼び出しで1枚生成）
- 生成された画像は微妙に異なるバリエーションとなる（API側の仕様）

**2.4.2.3 利用回数のカウント**
- 生成された画像1枚につき利用回数1回とカウント
- 例: 5枚生成した場合、利用回数は5回消費
- 生成前に利用可能回数をチェック:
  - 残り回数 < 生成指定枚数の場合: エラー表示「利用可能回数が不足しています（残り○枚）」
  - 生成をキャンセルし、枚数変更を促す

#### 2.4.3 画像変換の実行

**2.4.3.1 実行フロー**
1. ユーザーが画像をアップロード
2. プロンプトを選択または入力
3. 生成枚数を指定
4. 「変換開始」ボタンをクリック
5. バリデーション:
   - 画像未選択: 「画像を選択してください」
   - プロンプト未入力: 「プロンプトを入力してください」
   - 利用回数不足: 「利用可能回数が不足しています」
6. バリデーション通過後、変換処理開始

**2.4.3.2 変換処理（バックエンド）**
1. アップロードされた画像を一時保存
2. ユーザーの利用可能回数を確認
3. 画像とプロンプトをGemini 2.5 Flash Image API (Vertex AI) に送信
4. 指定枚数分、APIを順次呼び出し（キュー処理）
5. 生成された画像を保存（ユーザーごとに管理）
6. データベースに生成履歴を記録:
   - 元画像パス
   - 生成画像パス（複数）
   - 使用プロンプト
   - 生成枚数
   - 生成日時
   - 処理時間
7. ユーザーの利用済み回数を更新（+生成枚数）

**2.4.3.3 キュー処理**
- 複数ユーザーから同時にリクエストがあった場合、待ち行列（キュー）で管理
- キューの実装方法:
  - **推奨**: Django Celery + Redis/RabbitMQ
  - **軽量版**: Django-Q または django-rq
- 処理順序: FIFO（先入れ先出し）
- キュー待機中のユーザーへの通知: 「現在○件のリクエストが処理待ちです」（オプション）

### 2.5 リアルタイム進捗確認機能

#### 2.5.1 進捗表示仕様

**表示内容**:
- **処理開始時**: 「画像を変換しています...」
- **生成中**: 「画像 1/5 枚目を生成中...」「画像 2/5 枚目を生成中...」（動的更新）
- **完了時**: 「変換が完了しました！」
- **エラー時**: 「エラーが発生しました。もう一度お試しください。」

**UI要素**:
- ローディングアニメーション（スピナー）
- 進捗テキスト（上記メッセージ）
- 進捗インジケーター（例: 「2/5」の視覚的表示）

#### 2.5.2 技術実装

**実装方式: WebSocket + フォールバック（ポーリング）**

本システムでは、Django Channelsを使用したWebSocket実装を採用し、接続失敗時は自動的にポーリングにフォールバックする方式を実装しています。

**実装詳細**:
- **WebSocket**: Django Channels 4+ を使用
- **接続エンドポイント**: `/ws/conversion/{conversion_id}/`
- **認証**: Djangoセッション認証を自動適用
- **権限チェック**: 変換履歴の所有者のみ接続可能
- **フォールバック**: WebSocket接続失敗時、自動的に`GET /api/v1/convert/{conversion_id}/status/`をポーリング（4秒間隔）
- **再接続**: 最大5回、3秒間隔で再接続を試みます

**進捗通知の内容**:
- 処理開始時: 「画像変換を開始しています...」（進捗10%）
- API呼び出し前: 「AI画像生成中...」（進捗30%）
- 画像保存中: 「生成画像を保存中... (X/Y)」（進捗70-90%、各画像保存時に更新）
- 完了時: 「画像変換が完了しました！」（進捗100%）

**クライアント実装**:
- `static/js/websocket-client.js`: WebSocket接続管理ユーティリティ
- `static/js/progress.js`: 単一変換の進捗表示
- `static/js/progress_multiple.js`: 複数変換の同時監視

**メリット**:
- リアルタイム性: WebSocketで即座の更新
- 堅牢性: フォールバックにより接続失敗時も動作継続
- ユーザー体験: 進捗が細かく表示され、ユーザーが処理状況を把握しやすい

### 2.6 画像輝度調節機能

#### 2.6.1 機能概要
AI変換後の画像に対して、最終的な微調整として輝度（明るさ）を調整する機能

#### 2.6.2 詳細仕様

**2.6.2.1 調整タイミング**
- 画像変換完了後、ギャラリー画面または結果表示画面で調整可能
- リアルタイムプレビュー: スライダー操作で即座に反映

**2.6.2.2 UI要素**
- **スライダー**:
  - 範囲: -50（暗く） 〜 0（標準） 〜 +50（明るく）
  - デフォルト値: 0
  - ステップ: 1
- **プレビュー表示**: 調整中の画像をリアルタイム表示
- **リセットボタン**: スライダーを0に戻す
- **適用ボタン**: 調整を確定し、画像を保存

**2.6.2.3 処理方式**
- **推奨**: クライアント側（JavaScript）で画像を調整
  - ライブラリ: Pillow（Python側で最終保存時）、または Canvas API（ブラウザ側）
  - メリット: サーバー負荷なし、即座のフィードバック
- 「適用」ボタン押下時のみサーバーへ送信し、調整済み画像を保存

**2.6.2.4 調整ロジック**
- 輝度調整アルゴリズム: HSL色空間でLightness値を変更（推奨）
- 調整範囲: 一般的な画像編集ソフトと同等（±50%程度）

### 2.7 ギャラリー機能

#### 2.7.1 ギャラリー画面

**2.7.1.1 表示内容**
- ユーザーが過去に生成した画像の一覧
- サムネイル表示（グリッドレイアウト、推奨: 3〜4列）
- 各画像に以下の情報を表示:
  - サムネイル画像
  - 生成日時
  - 使用プロンプト（省略表示、クリックで全文表示）

**2.7.1.2 フィルタリング・ソート機能**
- **ソート**:
  - 新しい順（デフォルト）
  - 古い順
  - プロンプト名順（オプション）
- **検索**: プロンプトテキストでの部分一致検索（オプション）

#### 2.7.2 画像の詳細表示

**2.7.2.1 変換前後の比較表示**
- サムネイルクリックで詳細モーダルまたは専用ページを開く
- **表示方式**: 変換前後を横並びで表示
  - 左: 元画像（Before）
  - 右: 変換後画像（After）
- **表示情報**:
  - 生成日時
  - 使用プロンプト
  - 生成枚数（同一元画像から複数生成した場合、スライダーで切り替え）

**2.7.2.2 操作ボタン**
- **ダウンロードボタン**:
  - 個別画像のダウンロード
  - 形式: 元の形式（JPEG/PNG）を維持
  - ファイル名: `generated_YYYYMMDD_HHMMSS.jpg`（推奨）
- **削除ボタン**:
  - 確認ダイアログ表示「この画像を削除しますか？」
  - 削除実行後、ギャラリー一覧から即座に削除
  - サーバー側のファイルも削除

#### 2.7.3 画像の保存期間管理

**2.7.3.1 保存期間**
- **期間**: 生成日から30日間
- **自動削除**: 30日経過後、自動的にサーバーから削除
  - 実装方法: Cronジョブ（毎日1回実行）またはCelery Beatで定期実行
  - 削除対象: 生成日時が30日以前の画像ファイルとデータベースレコード

**2.7.3.2 保存容量制限**
- ユーザーごとの容量制限は設けない
- システム全体の容量監視は運用上推奨（アラート設定等）

### 2.8 利用回数制限機能

#### 2.8.1 制限の管理

**2.8.1.1 制限値の設定**
- **デフォルト値**: 100枚/月
- **個別設定**: 管理者が任意のユーザーの制限値を変更可能（管理画面から）
- **制限なし**: 9999枚など大きな値を設定することで実質無制限化

**2.8.1.2 利用回数のカウント**
- **カウント単位**: 生成された画像1枚につき1回
- **カウントタイミング**: 画像生成成功時
- **失敗時の扱い**: APIエラーなど失敗した場合はカウントしない

**2.8.1.3 制限のリセット**
- **リセットタイミング**: 毎月1日 00:00 (JST)
- **リセット処理**: 全ユーザーの「当月利用済み回数」を0にリセット
- **実装方法**: Cronジョブまたは Celery Beat で定期実行

#### 2.8.2 利用状況の表示

**2.8.2.1 ユーザー画面での表示**
- **表示位置**: ヘッダーまたはダッシュボード
- **表示内容**: 「今月の利用: ○○ / ○○○ 枚」
- **視覚的表示**: プログレスバー（オプション）
- **警告表示**: 残り10枚以下になったら「残りわずかです」等の注意喚起

**2.8.2.2 制限超過時の動作**
- 変換実行前にチェック
- 残り回数 < 生成枚数の場合:
  - エラーメッセージ: 「今月の利用可能回数を超過しています。残り○枚です。」
  - 変換処理は実行されない
  - 生成枚数を減らすか、管理者への連絡を促す

### 2.9 エラー処理・通知機能

#### 2.9.1 Gemini API エラー時の処理

**2.9.1.1 エラーの種類と対応**
- **APIレート制限エラー**:
  - ユーザーへの通知: 「現在アクセスが集中しています。しばらく待ってから再度お試しください。」
  - 対応: キューに入れて再試行（推奨: 3回まで、指数バックオフ）
  
- **API認証エラー**:
  - ユーザーへの通知: 「システムエラーが発生しました。管理者にお問い合わせください。（エラーコード: AUTH001）」
  - ログに記録（サーバーログ）
  
- **画像処理エラー**:
  - ユーザーへの通知: 「画像の処理中にエラーが発生しました。別の画像でお試しください。」
  
- **タイムアウトエラー**:
  - ユーザーへの通知: 「処理に時間がかかりすぎています。もう一度お試しください。」

**2.9.1.2 通知方式**
- **推奨方式**: トースト通知（画面右上に数秒間表示されるポップアップ）
- **代替方式**: アラートダイアログ、インラインエラーメッセージ
- **色分け**: エラー（赤）、警告（黄）、成功（緑）

**2.9.1.3 利用回数のカウント**
- **成功時のみカウント**: エラーが発生した場合、利用回数はカウントしない
- **リトライ時**: リトライで成功した場合のみカウント

#### 2.9.2 その他のエラー処理

**2.9.2.1 ファイルアップロードエラー**
- ファイルサイズ超過
- 対応していない形式
- ネットワークエラー
→ 各エラーに応じた具体的なメッセージを表示

**2.9.2.2 ネットワークエラー**
- オフライン状態での操作
- サーバーダウン
→ 「インターネット接続を確認してください」等のメッセージ

**2.9.2.3 管理者へのエラー通知**
- 不要（ログファイルでの確認を想定）
- オプション: 重大なエラー（API認証失敗等）はログレベルをERRORに設定し、外部監視ツール連携を検討

## 5. API仕様（内部API）

### 5.1 画像アップロードAPI
- **エンドポイント**: `POST /api/upload/`
- **リクエスト**: multipart/form-data (画像ファイル)
- **レスポンス**: 
  ```json
  {
    "success": true,
    "image_id": "12345",
    "preview_url": "/media/uploads/preview.jpg"
  }
  ```

### 5.2 画像変換開始API
- **エンドポイント**: `POST /api/convert/`
- **リクエスト**:
  ```json
  {
    "image_id": "12345",
    "prompt": "背景を白に変更",
    "generation_count": 3
  }
  ```
- **レスポンス**:
  ```json
  {
    "success": true,
    "job_id": "job_67890",
    "message": "変換処理を開始しました"
  }
  ```

### 5.3 進捗確認API
- **エンドポイント**: `GET /api/progress/{job_id}/`
- **レスポンス**:
  ```json
  {
    "status": "processing",
    "current": 2,
    "total": 5,
    "message": "画像 2/5 枚目を生成中..."
  }
  ```

### 5.4 ギャラリー取得API
- **エンドポイント**: `GET /api/gallery/`
- **レスポンス**:
  ```json
  {
    "images": [
      {
        "id": "img_123",
        "original_image": "/media/original.jpg",
        "generated_images": ["/media/gen1.jpg", "/media/gen2.jpg"],
        "prompt": "明るく爽やか",
        "created_at": "2025-10-30T10:30:00Z"
      }
    ]
  }
  ```

## 7. セキュリティ要件

### 7.1 通信のセキュリティ
- **HTTPS必須**: 全ての通信をHTTPSで暗号化
- **SSL証明書**: Let's Encrypt等の信頼できる証明書を使用

### 7.2 認証・認可
- **パスワード保存**: ハッシュ化（Django標準のPBKDF2）
- **セッション管理**: Django標準のセッションフレームワーク使用
- **CSRF対策**: Django標準のCSRFトークン使用

### 7.3 データ保護
- **ファイルアクセス制限**: ユーザーは自分の画像のみアクセス可能
- **パス・トラバーサル対策**: ファイルパスの検証
- **SQLインジェクション対策**: Djangoのクエリセット使用（パラメータバインド）
- **XSS対策**: テンプレートエンジンの自動エスケープ機能使用

### 7.4 API認証情報の管理
- **環境変数**: Vertex AI認証情報は環境変数で管理（`.env`ファイル）
- **バージョン管理から除外**: `.gitignore`に認証情報を記載

### 8.4 バックアップ
- **データベース**: 日次自動バックアップ（推奨）
- **画像ファイル**: 定期バックアップ（オプション）

## 9. 制約事項・前提条件

### 9.1 制約事項
- ユーザー自身によるパスワード変更機能は提供しない
- ログイン失敗によるアカウントロック機能は実装しない
- 管理者へのエラーメール通知機能は実装しない
- ユーザーごとの保存容量制限は設けない

### 9.2 前提条件
- ユーザーはモダンブラウザ（Chrome, Firefox, Safari, Edge最新版）を使用
- インターネット接続環境がある
- 管理者はDjango Adminの基本操作を理解している
- Google Cloud Platformのアカウントと Vertex AI の有効化が完了している
